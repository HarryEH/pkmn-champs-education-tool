## Overview

An Electron Forge + Vite + React 19 + TypeScript desktop app for **Pokémon Champions VGC**
(`gen9championsvgc2026regma`, `CURRENT_FORMAT` in `src/shared/types.ts`). It has three flows,
all implemented:

- **Team Setup** — import your team via PokePaste/Showdown export, validated for Champions
  Reg M-A legality (species/item/ability/move/learnset), persisted to disk.
- **Detection** — drop a Nintendo Switch team-preview screenshot, identify the 6 opponent icons
  via CLIP image-embedding nearest-neighbour, then show per-opponent common sets, type matchups,
  speed tiers, and damage calcs vs. your active team.
- **In-Battle** — narrow Detection's output to the active 4v4 (and the up-to-2 currently on
  field per side), with field-state and per-mon Mega/Tera toggles for live speed/damage recalc.

**Build/test health (verified now):** `tsc --noEmit` clean · `eslint` 8 warnings (non-null
assertions in test files only, 0 errors) · `vitest` **103/103 passing across 15 files** ·
detection accuracy gate **5/6 top-1** on the real Jason frame.

## What's implemented (verified against the code)

### Process model

- **Main** (`src/main.ts`, `src/main/ipc/persistence.ts`, `src/main/media.ts`): 1280×800 window,
  DevTools in dev only, macOS camera-permission prompt, and hand-rolled JSON persistence in
  `app.getPath('userData')`. No Pokémon/calc logic.
- **Preload** (`src/preload.ts`): the only sanctioned IPC surface — a typed `window.api` bridge
  via `contextBridge` (`teams`, `settings`, `usage`, `media`).
- **Renderer**: everything else — all `@pkmn`/`@smogon` calc, detection (canvas + CLIP), and the
  React UI.

### Shared contracts (`src/shared/`)

- `types.ts` (**FROZEN**): `MyTeam`/`MyPokemon` (only persisted domain data), `OpponentTeam`/
  `OpponentSlot`, `FieldState`/`SideState`/`BattleSession`, `Settings`, `UsageData`/`SpeciesUsage`/
  `UsageEntry`, `NormalizedRect`, `CURRENT_FORMAT`.
- `ipc.ts` (**FROZEN**): channel constants + the `Api` interface (`teams.*`, `settings.*`,
  `usage.*`, `media.requestCamera`).
- `fixtures.ts`: `FIXTURE_MY_TEAM` + `FIXTURE_OPPONENT_TEAM` — used as live fallbacks throughout
  the Detection and In-Battle screens so they work before any team is imported or opponent detected.

### Stores (Zustand, `src/renderer/store/`)

- `teams.ts` — persisted (write-through to `window.api.teams`). Owns `parsePokepaste`/`createTeam`/
  `computeStat`, CRUD, active-team selection, and the **non-blocking legality** import errors.
- `settings.ts` — persisted. Capture device id, calibration rects, theme mode.
- `session.ts` — **in-memory only**, reset on "New Battle"/restart. Holds detected `OpponentTeam`,
  `FieldState`, `myActiveFour`/`opponentActiveFour` (brought 4), `myOnField`/`opponentOnField`
  (currently in, cap 2), and `myBattleState` (your-side per-mon Mega/Tera toggles, kept off the
  FROZEN `types.ts`).
- `nav.ts` — screen-enum routing (no router).

All persisted stores hydrate from IPC on boot.

### The `gen` singleton (`src/lib/calc/gen.ts`)

Single source of all Pokémon data and calc. **It is ungated** —
`new Generations(Dex, () => true).get(9)` — so it resolves the full National-Dex pool including
Mega Stones, Mega formes, and never-released bases (e.g. Floette-Eternal) that Champions revives.
Gen 9 is correct for *mechanics* (stat/damage formulae, type chart); legality is layered on top
from baked tables, not from `@pkmn/dex`. **Consequence:** a miss returns an object with
`exists === false` (not `undefined`) — callers must test `.exists`.

Imports `@smogon/calc` from `@smogon/calc/dist/adaptable` (the `/adaptable` path doesn't resolve —
`@smogon/calc@0.11` ships no `exports` map); re-exports `calculate`/`Pokemon`/`Move`/`Field`.

### Team Setup (`src/renderer/screens/TeamSetup/`, `store/teams.ts`)

PokePaste textarea → split on blank lines → `Sets.importSet` → resolve vs `gen.species` → live
gallery of `PokemonCard`s (icon via `@pkmn/img`, stats with **Speed emphasised**). Save → persist →
active-team picker → edit/delete/re-paste-to-update. Each resolved set is run through full
Champions legality (see below); violations surface as **non-blocking** `ImportError`s (the Pokémon
stays in the gallery).

### Detection pipeline — CLIP box-embedding nearest-neighbour (`src/lib/detection/`)

Opponent identification is CLIP image-embedding NN against pokesprite box sprites (the blockhash
pipeline was retired — see "What changed"):

1. `imageSource.ts::loadImageFromFile` (dropped screenshot) → `RgbaImage` (shared type in
   `image.ts`). The pipeline is source-agnostic, so a future live-capture path can produce the
   same `RgbaImage` from a `<video>` element and reuse detection unchanged.
2. `cropRegions.ts` slices the frame into 6 crops from normalized `NormalizedRect[]` (calibration).
3. `segment.ts::segmentToWhite` removes the red opponent-panel background — the **default,
   primary** preprocessing in the pipeline (`detectionPipeline.ts` applies it unless
   `skipSegmentation` is passed).
4. `embedder.ts::embedCrop` lazily runs CLIP ViT-B/32 (`@huggingface/transformers`,
   `Xenova/clip-vit-base-patch32`, downloaded on first run + browser-cached for offline reuse)
   over `compositeOnWhite(crop)` → a raw 512-d vector. `embedPreproc.ts` owns the composite step
   (build/runtime parity via `PREPROC_VERSION`).
5. `iconMatcher.ts::matchEmbedding` mean-centers the crop with the table's stored pool mean,
   cosine-ranks against the **legal-only** entries → top-3 + confidence; `AUTO_ACCEPT_THRESHOLD`
   (0.7) + `AUTO_ACCEPT_MARGIN` (0.03) decide auto-confirm vs. manual override.
6. `detectionPipeline.ts::detectOpponentTeam` (async) wires 2–5 into an `OpponentTeam`.

`boxEmbeddings.ts` is the single source of truth for the table shape + centering/cosine math, and
asserts model/preproc parity. The regression gate is `__tests__/detectionAccuracy.test.ts` (real
Jason frame, headless via precomputed crop embeddings).

The **Detection screen** (`screens/Detection/`) is the **screenshot-drop path (E1a)**: drop/pick a
file, calibrate 6 boxes over a `CalibrationOverlay` (pre-seeded defaults, save/reset), "Detect" →
`SlotList` with per-slot confidence + override dropdowns, then the full `OpponentDashboard`. The
model is preloaded on screen mount with a "Loading model…" state. **Live capture (E1b) is
unbuilt** — no renderer code calls `getUserMedia`/`enumerateDevices`, and the earlier orphaned
`frameCapture.ts` stub was removed in the 2026-06-15 cleanup (to be rebuilt when E1b is tackled).

### Legality module (`src/lib/legality/`) — not mentioned in CLAUDE.md/old docs

A full team-legality layer beyond just species:

- `teamLegality.ts::checkSetLegality(set, species)` — one synchronous entry point combining five
  layers: species (`championsLegality.json`), item/ability/move bans (`championsOverrides.json`
  delta + vanilla `gen.{items,abilities,moves}`), and per-species learnsets
  (`championsLearnsets.json`, pre-merged). Returns human-readable violation strings.
- `championsOverrides.ts` — `isItemLegal`/`isMoveLegal`/`isAbilityLegal` over the override delta.
  The critical correctness rule: `isNonstandard: null` is an explicit *un-ban* and must round-trip
  distinctly from "field absent" (uses `'isNonstandard' in override` presence checks, never `??`).
- `championsLearnsets.ts` — `buildLearnsetIndex`/`canLearnMove` over the baked movepool table
  (baked because `gen.learnsets.get()` is async and `parsePokepaste` is sync).
- Species legality itself lives in `src/lib/detection/championsLegality.ts`
  (`deriveLegality`/`buildLegalityIndex`/`isChampionsLegal`), shared by detection and Team Setup.

### Calc (`src/lib/calc/`)

- `damageCalc.ts` — `calcDamage(attacker, defender, move, field)` over `calculate()`; builds
  `Pokemon` from either a `SetCombatant` (parsed set) or a `SpeciesCombatant` (speciesId + usage
  overrides), maps `FieldState`→`Field` (always `gameType: 'Doubles'`), supports `megaActivated`/
  explicit `megaForme`. Returns min/max %, KO chance, and the calc's description string.
- `speedTiers.ts` — `calcSpeed`, composable `applySpeedModifiers` (Tailwind ×2 / Choice Scarf ×1.5
  / paralysis / Trick Room), `buildSpeedTiers`, and `speedBounds` (0-EV neutral → max-invest, for
  the unknown opponent spread range).
- `typeMatchup.ts` — `getMatchup`/`defensiveProfile` pure lookups from `gen.types`.
- `megaForme.ts` — `resolveMegaForme(species, item)` (held stone → forme via `gen.items` `megaStone`
  record), `megaFormesOf`, `defaultMegaForme` (fallback forme when the opponent's stone is unknown).

### Smogon usage data (`src/lib/smogon/usageData.ts`)

`fetchUsage(format, {refresh, now, fetchImpl})` fetches `https://data.pkmn.cc/stats/<format>.json`
directly from the renderer, shapes it into `UsageData`/`SpeciesUsage` (items/abilities/teraTypes/
moves/spreads, each sorted descending by usage), and read-through caches via `window.api.usage`.
Endpoint serves "latest" only; the month key partitions the disk cache. Offline/404 degrades to
cached or an empty-but-valid `UsageData`, never throws. (`gen9championsvgc2026regma` currently 404s
upstream — Reg M-A just landed — so the dashboard renders with empty usage today.)

### Usage-driven analysis dashboards (Detection E2 + In-Battle)

This is the part the old docs under-described and is directly relevant to the roadmap below — **it
already exists in a first-pass form:**

- **Detection** `OpponentDashboard.tsx` + `opponentBuild.ts`: one tab per opponent. Each tab shows
  **common sets** (top items/abilities/Tera/moves/spreads by usage %), a `TypeMatchupGrid`,
  a `SpeedTierList` (your 6 vs. the opponent's usage-derived speed stat), and two `DamageCalcTable`s
  — your damaging moves vs. the opponent, and the opponent's **usage-top moves** (`topMoves(usage, 4)`)
  vs. your team. `buildOpponentCombatant` derives a representative set from the opponent's most-used
  item/ability/Tera/spread.
- **In-Battle** `index.tsx` + `battleBuild.ts`: two-step on-field selection per side, per-mon
  Mega/Tera toggles, `FieldStateToggles`. Speed order + both damage matrices recompute live,
  restricted to on-field mons, with the opponent shown as a **speed range** (min / max / max+Scarf)
  and their attacking rows driven by `topMoves(usage, 4)`.

### UI/theme

`ui/` design-system primitives (Button, Card, Tabs, Toggle, Select, TypeBadge, Stat) + `ui/Gallery.tsx`
showcase; `theme/tokens.css`, `theme/types.ts` (18 official type colours), `theme/matchup.ts`
(`matchupTint`). Sprites/icons always via `@pkmn/img` (`components/pokemonIcon.ts`). Type colours are
always paired with text labels.

## What still needs to be done (prioritized roadmap)

The product owner's two stated priorities are **already implemented in first-pass form** (see the
"usage-driven analysis dashboards" section). The work below is therefore mostly *deepening* what
exists, plus genuine gaps found in the code.

### Priority 1 — Common sets / moves / items for the opponent (deepen)

*Status: first pass shipped.* The Detection dashboard already surfaces top items/abilities/Tera/
moves/spreads per opponent (`OpponentDashboard.tsx` `CommonSets`), and `buildOpponentCombatant`
derives a representative set. Remaining work:

1. The live format `gen9championsvgc2026regma` **404s upstream**, so today the dashboard renders
   empty usage. Re-point/confirm `fetchUsage` once Champions stats publish (and verify the
   name-vs-id keying in `findUsage` against real Champions data).
2. Surface **multiple distinct sets** rather than a single most-common composite (the current
   `buildOpponentCombatant` picks the top of each axis independently, which can synthesize a set no
   real player runs). Consider per-set move/item bundles.
3. There is no manual "I saw their item/ability" entry from common-set context yet beyond the
   In-Battle slot overrides — wire revealed-info back into the displayed set.

### Priority 2 — In-battle speed tiers + damage from the opponent's most-likely moves (deepen)

*Status: first pass shipped.* In-Battle already drives both damage matrices off `topMoves(usage, 4)`
and shows the opponent speed as a min/max/+Scarf range, recomputing live on every toggle. Remaining
work:

1. **Move selection is "top 4 by usage," not "the relevant attacking moves."** It does not filter
   to damaging moves, weight by matchup, or pick the threatening move per defender. Make the
   opponent-move set matchup-aware (e.g. STAB/super-effective coverage vs. each of your on-field
   mons) rather than raw usage order.
2. Speed range is coarse (0-EV neutral → max-invest + Scarf). Consider folding the **common usage
   spread**'s actual Speed EVs in as the most-likely line, alongside the bounds.
3. The empty-usage 404 (above) blanks the opponent move rows entirely in-battle — same dependency
   as Priority 1.1.

### Priority 3 — Detection accuracy & calibration

4. **Slot-3 (Rotom-Wash) miss.** Rotom appliance formes are not in the top 3 and the top cluster is
   within 0.001 — improve box-sprite references for Rotom formes / similar small renders, or add a
   tie-break/disambiguation step. Collect more real frames to tune `AUTO_ACCEPT_THRESHOLD`/`MARGIN`.
5. **Calibration UX** relies on hand-positioned default rects for one Switch resolution; there is no
   auto-detection of the icon row, and no per-resolution presets. Validate/extend defaults across
   capture resolutions.

### Priority 4 — Live capture (E1b) and polish

6. **Live capture device is unbuilt.** Nothing calls `getUserMedia`/`enumerateDevices`. Add device
   enumeration + picker (persist `deviceId`), a `<video>` element, and a small capture helper that
   produces an `RgbaImage` to feed into the same `detectionPipeline` call the screenshot path uses
   (the removed `frameCapture.ts` stub did roughly this — rebuild it then).
7. **Battle-mode theme toggle** (`themeMode` exists in `Settings` but isn't surfaced in chrome).

### Priority 5 — Reg M-A → M-B cutover (release checklist, dated 2026-06-17)

8. Regenerate the three regulation-specific tables and bump `CURRENT_FORMAT`:
   `npx vite-node scripts/buildChampionsLegality.ts`,
   `scripts/buildChampionsOverrides.ts`, `scripts/buildChampionsLearnsets.ts`. Re-point `fetchUsage`
   once `gen9championsvgc2026regmb` stats publish. `boxEmbeddings.json` does **not** need
   regeneration (regulation-independent; legal-only filtering is at runtime).

### Known limitation (carry-over, not a bug)

- `@smogon/calc` does not auto-Mega-evolve a base species from its held stone — the Mega forme name
  must be passed explicitly. The code handles this via `resolveMegaForme`/`megaForme`, but the
  opponent Mega toggle only auto-surfaces when usage data resolves a stone; a default forme is used
  otherwise.
